# H4KKEN Architecture

## Stack

- **Runtime / package manager**: Bun
- **Bundler**: Vite
- **Renderer**: Babylon.js 8
- **Language**: TypeScript with strict checking
- **Validation**: `bun run fix`, `bun run typecheck`, `bun run tests/net-sim/run-sim.ts`

## Runtime model

H4KKEN is a client-rendered fighting game with a lightweight relay server.

- The browser runs rendering, local simulation, rollback, input capture, UI, and audio.
- The server handles static assets, matchmaking, room lifecycle, WebSocket relay, WebRTC signaling relay, health checks, and optional TURN credentials.
- Online matches always start on WebSocket and try to upgrade the binary input path to WebRTC DataChannel in the background.
- If WebRTC fails, gameplay continues on WebSocket without interrupting the match.

## Networking

- **WebSocket** carries matchmaking, room control, countdown flow, round results, ping, and WebRTC signaling.
- **WebRTC DataChannel** carries binary input frames with unordered, no-retransmit semantics.
- **TURN** is optional and expected to be self-hosted coturn using HMAC auth (`use-auth-secret` / `static-auth-secret`).
- **Fallback behavior** is intentional: no TURN means STUN-only attempt, and any WebRTC failure leaves the game on WebSocket.

### Server endpoints

- `GET /health` returns uptime, rooms, waiting players, and connections.
- `GET /api/turn-credentials` returns coturn credentials when `TURN_SECRET` and `TURN_REALM` are configured; otherwise it returns `stun-only`.

## Rollback model

- Simulation runs at 60 Hz.
- Both clients simulate locally and exchange compact input packets.
- Remote inputs are predicted until confirmed.
- On mismatch, the game restores a snapshot and replays deterministically.
- Practice mode bypasses rollback and uses a local bot.

## Rendering and assets

- Babylon.js uses the default left-handed, Y-up coordinate system.
- Fighters are animated from shared GLB assets and cloned skeletons.
- Static stage content is aggressively frozen for performance.
- Remote fighters use visual smoothing to hide small rollback corrections.

## Diagnostics and testing

- Press `F3` to toggle the network diagnostics overlay during matches.
- `tests/net-sim/` is a local, headless harness for rollback behavior under simulated latency, jitter, and packet loss.
- The net-sim harness validates determinism and rollback behavior, not real TURN, ICE, browser WebRTC, or deployment wiring.

## Operational notes

- The server is intentionally usable with zero env vars.
- coturn integration must match the credentials generated by `server.ts`: same realm, HMAC shared secret, public listener IP, and open relay UDP port range.
- Keep this file brief and current; the code remains the primary source of truth.

## Adding a new character

### Source files

Place the source in `assets/source/<id>.blend` (preferred) or `<id>.glb`/`<id>.fbx`. The blend file is the editable source of truth; export FBX or GLB from it before building.

For GLB sources (e.g. Mixamo characters): `export_mesh.py` bakes the rest pose and normalises the armature scale to 1.0. Raw Mixamo GLBs have `scale=0.01` on the armature — never skip this step or the character will be tiny and lie flat.

For emissive materials: set the emissive color to a saturated hue at strength ≥ 1.0 in Blender rather than a near-white at low strength. The pipeline preserves `emissiveFactor` from the GLB and boosts it at runtime — a white-ish color will blow out.

### Build pipeline

1. Register in `scripts/characters.ts` — source path, ground corrections (set all to `0` initially).
2. Run `bun run build:character --char <id> --force` to produce `public/assets/models/<id>.glb`.
3. Compute ground corrections: run `scripts/compute_ground_corrections.py`, add the output values to `scripts/characters.ts`.
4. Rebuild with corrections: `bun run build:character --char <id> --force`.

### Runtime registration

Add an entry to `src/fighter/characters.ts`:

```ts
<id>: {
  id: '<id>',
  name: 'Display Name',
  scale: 1.0,           // adjust until height matches beano visually
  glowEmissive: true,   // only if the character has emissive materials
}
```

- `scale` — after the bake the armature is `scale=1.0`; tune runtime scale for desired height.
- `glowEmissive` — enables the emissive boost (×20, clamped per channel) at load time so the GlowLayer in char select picks it up.
- `jiggleBones` — optional spring-bone secondary motion; see mita's entry for the pattern.

### Ground corrections sign convention

A **negative** value means the character sinks into the floor → the correction lifts the hips up. A **positive** value means the character floats → the correction pushes down. `compute_ground_corrections.py` outputs the correct sign — paste values as-is.
